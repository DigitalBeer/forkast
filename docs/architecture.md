# Meal Planning Application Full-Stack Architecture Document

**Version:** 1.0
**Date:** July 9, 2025

## Section 1: Introduction

This document outlines the complete full-stack architecture for the Meal Planning Application, based on a foundation of Next.js and Supabase, to be housed in a monorepo.

## Section 2: High Level Architecture

The application uses a Backend-as-a-Service (BaaS) model where the Next.js frontend (on Vercel) communicates directly with Supabase for all backend services (Auth, Database, Storage, Edge Functions).

## Section 3: Tech Stack

The tech stack is built around TypeScript, Next.js/React, Tailwind CSS, and Shadcn/ui on the frontend, with Zustand for state management. The backend is the full Supabase suite. Testing will be done with Jest, React Testing Library, and Playwright. CI/CD will be managed by GitHub Actions.

## Section 4: Data Models

The database schema is defined to support profiles, meals, meal plans, and the history of planned meals, including subscription status.

## Section 5: Components

The application is broken into three main components within a monorepo: `web` (the Next.js frontend), `supabase` (the BaaS platform configuration), and `shared` (for common TypeScript types).

## Section 6: External APIs

No external third-party APIs are required for the MVP.

## Section 7: Core Workflows

Sequence diagrams have been defined for both anonymous and registered user flows, detailing the interactions between the frontend, browser local storage, and the Supabase backend.

## Section 8: API Specification

The API consists of the auto-generated PostgREST API from Supabase for CRUD operations and a custom `POST /functions/v1/get-meal-suggestions` endpoint for the recommendation engine.

## Section 9: Database Schema

```sql
CREATE TYPE public.subscription_status AS ENUM ('free', 'premium');

CREATE TABLE public.profiles (
  id uuid NOT NULL PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  full_name text,
  subscription_status public.subscription_status DEFAULT 'free' NOT NULL,
  stripe_customer_id text,
  subscription_current_period_end timestamptz,
  created_at timestamptz DEFAULT now()
);

CREATE TABLE public.meals (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id uuid NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  name text NOT NULL,
  meal_type text NOT NULL,
  image_url text,
  source_url text,
  ingredients text,
  instructions text,
  rating smallint CHECK (rating >= 1 AND rating <= 5),
  dietary_tags text[],
  created_at timestamptz DEFAULT now()
);
CREATE INDEX ON public.meals (user_id);

CREATE TABLE public.meal_plans (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id uuid NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  start_date date NOT NULL,
  end_date date NOT NULL,
  created_at timestamptz DEFAULT now()
);
CREATE INDEX ON public.meal_plans (user_id);

CREATE TABLE public.planned_meals (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  meal_plan_id bigint NOT NULL REFERENCES public.meal_plans(id) ON DELETE CASCADE,
  meal_id bigint NOT NULL REFERENCES public.meals(id) ON DELETE CASCADE,
  planned_for_date date NOT NULL,
  action_type text NOT NULL,
  created_at timestamptz DEFAULT now()
);
CREATE INDEX ON public.planned_meals (meal_plan_id);
CREATE INDEX ON public.planned_meals (meal_id);
```

## Section 11: Infrastructure and Deployment

A CI/CD pipeline using GitHub Actions will manage automated deployments to Vercel and database migrations via the Supabase CLI. Preview deployments will be created for every pull request, and merges to the main branch will deploy to production.

## Section 12: Error Handling Strategy

A centralized strategy using structured JSON logs, global error handlers, and database transactions will be employed to ensure application reliability.

## Section 13: Coding Standards

A minimal, mandatory set of standards for linting, formatting, naming, and critical patterns has been defined to ensure code quality and consistency for AI agents.

## Section 14: Test Strategy and Standards

A pragmatic testing strategy focusing on Integration and E2E tests for critical user flows and business logic has been established, with automated execution in the CI/CD pipeline.

## Section 15: Security

Security relies on Supabase Auth with Row-Level Security (RLS) policies on all user data tables. Secrets are managed via environment variables in GitHub/Vercel, and Dependabot is used for dependency scanning.
